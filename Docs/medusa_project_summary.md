# MedusaJS 2.0: Технический план запуска и интеграции

> [!summary] Суть документа
> Краткий технический план по запуску проекта на **MedusaJS 2.0**: стек, архитектура процессов, этапы разработки, деплой и роль AI в ускорении реализации.

## 1. Обзор платформы

MedusaJS — это **headless e-commerce** пла2тформа на базе Node.js, где бэкенд и витрина разделены архитектурно. Такой подход упрощает масштабирование, кастомизацию бизнес-логики и независимую разработку frontend/backend-частей.

> [!info] Ключевое преимущество
> Основная ценность Medusa — модульная архитектура и изоляция доменной логики от пользовательского интерфейса.

### Базовые системные требования

| Параметр | Рекомендация | Комментарий |
| --- | --- | --- |
| CPU | `2 vCPU` | Минимальный рабочий порог |
| RAM | `4 GB` | Достаточно для Node.js, PostgreSQL и Redis на одном VPS |
| Runtime | `Node.js 20+` | Базовая среда исполнения |
| Database | `PostgreSQL 14+` | Основное хранилище |
| Cache / Queue | `Redis 6.2+` | Очереди, кеш и фоновые задачи |
| Hosting | `VPS + Ubuntu 22.04+` | Предпочтительно с Docker-изоляцией |

## 2. Архитектура запуска: `shared` / `server` / `worker`

Medusa 2.0 поддерживает несколько режимов выполнения, что позволяет начать с простой схемы и перейти к разделению нагрузки без смены платформы.

| Режим | Конфигурация | Назначение | Когда использовать |
| --- | --- | --- | --- |
| `Shared` | По умолчанию | Один процесс обслуживает API и фоновые задачи | Старт проекта, малый бизнес, минимальная инфраструктура |
| `Server` | `MEDUSA_WORKER_MODE=server` | Обрабатывает только API-запросы | Когда нужен быстрый отклик интерфейса |
| `Worker` | `MEDUSA_WORKER_MODE=worker` | Выполняет фоновые задачи | Рассылки, платежи, синхронизации, уведомления |

> [!tip] Практический выбор
> Для первого запуска достаточно режима **Shared**. Разделение на `server` и `worker` имеет смысл после появления заметной фоновой нагрузки.

## 3. Дорожная карта запуска

> [!todo] Принцип запуска
> Каждая фаза должна завершаться не просто кодом, а **проверяемым результатом**: рабочим сервисом, пройденным сценарием или готовым артефактом для следующего этапа.

### Карта этапов

```text
Фаза 0 -> Фаза 1 -> Фаза 2 -> Фаза 3 -> Фаза 4 -> Фаза 5
Подготовка   Backend    Auth/Comms   Storefront   Events/VK   Pre-Launch
```

| Фаза | Фокус | Ключевой результат | Что разблокирует |
| --- | --- | --- | --- |
| `0` | Подготовка среды | Репозиторий, Docker-окружение, базовая конфигурация | Начало backend-разработки |
| `1` | Core backend | Рабочий Medusa backend с БД и Redis | Подключение auth и frontend |
| `2` | Авторизация и коммуникации | VK login + email как резервный канал | Личный кабинет и customer flow |
| `3` | Storefront | Пользовательская витрина с checkout | Тестирование реального e-commerce потока |
| `4` | События и уведомления | Отправка уведомлений по событиям заказа | Полный lifecycle заказа |
| `5` | Предзапуск и деплой | Проверенный релизный контур | Production rollout |

### Фаза 0. Подготовка среды и каркаса проекта

**Цель:** быстро собрать воспроизводимую локальную среду, на которой можно безопасно строить всю остальную систему.

| Блок | Что сделать | Артефакт на выходе |
| --- | --- | --- |
| Репозиторий | Создать проект через `npx create-medusa-app@latest` | Базовый проект Medusa |
| Инфраструктура | Поднять `PostgreSQL` и `Redis` в Docker | Локальное окружение в один запуск |
| Конфигурация | Подготовить `.env`, переменные подключения, базовые секреты | Предсказуемая конфигурация окружения |
| AI-помощник | Подключить **Claude AI / Claude Code** с `medusa-dev` | Ускорение генерации модулей и миграций |

**Контрольные точки**

- backend стартует без ошибок;
- Medusa подключается к БД и Redis;
- команда запуска понятна и воспроизводима для нового разработчика.

> [!note] Go / No-Go
> Переходить дальше имеет смысл только после стабильного локального старта без ручных «докруток» после каждого `docker compose up`.

### Фаза 1. Core backend и доменная база

**Цель:** получить backend, готовый к расширению бизнес-логики, а не только «запущенный шаблон».

| Направление | Задачи | Выход |
| --- | --- | --- |
| Базовые модули | Проверить стандартные сущности: продукты, регионы, заказы, клиенты | Понятная доменная база |
| Миграции | Настроить рабочий процесс через `npx medusa db:migrate` | Контролируемые изменения схемы БД |
| Сидирование | Добавить стартовые данные для разработки | Быстрый smoke test storefront/API |
| API-проверка | Прогнать базовые сценарии через Admin/API endpoints | Подтверждение корректной работы backend |

**Что должно быть готово**

- создаются и читаются товары;
- доступен admin/backend API;
- миграции применяются без ручного вмешательства;
- dev-окружение можно пересобрать с нуля.

> [!tip] Практический фокус
> На этой фазе лучше не распыляться на кастомные функции. Сначала нужен стабильный фундамент: схема, данные, базовые API и повторяемая сборка.

### Фаза 2. Авторизация и коммуникационные каналы

**Цель:** собрать устойчивый customer entrypoint, где VK является основным способом входа, а email закрывает резервный сценарий.

| Подсистема | Задачи | Критерий готовности |
| --- | --- | --- |
| `VK OAuth` | Зарегистрировать приложение в `VK Console`, подключить `passport-vkontakte`, настроить callback | Пользователь может войти через VK |
| Customer mapping | Если `VK ID` не найден, создать `Customer`; если найден, выдать `JWT` | Нет дублей пользователей |
| Email provider | Подключить `SendGrid`, `Resend` или `SMTP` | Письма отправляются из dev/stage среды |
| Email templates | Подготовить `Welcome email` и `Order confirmation` | Базовые транзакционные письма готовы |

**Ключевые сценарии**

1. Новый пользователь заходит через VK и автоматически получает customer-профиль.
2. Существующий пользователь логинится повторно без создания дубля.
3. После ключевых действий система может отправить email как резервное уведомление.

> [!warning] Основной риск
> Ошибки на этом этапе обычно связаны не с Medusa, а с callback URL, токенами VK и неконсистентной логикой связывания `VK ID <-> Customer`.

### Фаза 3. Storefront на Next.js

**Цель:** перевести backend из режима «API готово» в режим «пользователь реально может купить товар».

| Зона | Что реализовать | Результат |
| --- | --- | --- |
| Основа витрины | Взять официальный **Next.js Starter** и подключить SDK | Рабочая витрина поверх Medusa |
| Data layer | Интегрировать `TanStack Query` для кеширования и повторных запросов | Быстрый и предсказуемый клиентский слой |
| SEO | Настроить `Metadata`, `OpenGraph`, `Sitemap` | Корректная индексация и красивый шаринг |
| Покупательский путь | Каталог, карточка товара, корзина, checkout | Полноценный путь до оформления заказа |
| Customer area | Личный кабинет и история заказов | Повторные пользователи видят свой контур |

**Definition of Done**

- пользователь может открыть каталог, добавить товар в корзину и дойти до checkout;
- storefront корректно работает с backend API;
- ссылки для VK и поисковых систем имеют валидный preview;
- основные страницы читаемы и на desktop, и на mobile.

> [!check] Ключевой рубеж
> После завершения этой фазы проект перестает быть «технической сборкой» и становится продуктом, который уже можно показывать и тестировать на реальных сценариях.

### Фаза 4. События и уведомления через VK

**Цель:** автоматизировать post-order коммуникацию и сделать события заказа видимыми для клиента.

| Компонент | Что сделать | Проверка |
| --- | --- | --- |
| VK bot | Создать бота сообщества и подготовить канал отправки | Бот может отправить тестовое сообщение |
| Notification Provider | Реализовать кастомный `Notification Provider` | Medusa умеет вызывать провайдер |
| Event subscriptions | Подписаться на `order.placed`, `order.shipped` | События реально срабатывают |
| Delivery logic | Отправлять сообщение через `messages.send` | Клиент получает уведомление в VK |

**Минимальный набор событий**

- `order.placed`
- `order.shipped`

**Желательно добавить после MVP**

- `order.payment_failed`
- `order.canceled`
- напоминания о незавершенном checkout

> [!success] Бизнес-эффект
> Эта фаза превращает заказ из «записи в системе» в управляемый lifecycle с прозрачной коммуникацией для клиента.

### Фаза 5. Предзапуск, стабилизация и выпуск

**Цель:** не просто развернуть сервис, а довести систему до состояния безопасного production-запуска.

| Направление | Что проверить | Результат |
| --- | --- | --- |
| Контейнеризация | Собрать production-образ через `Dockerfile` | Готовый релизный контейнер |
| База данных | Прогнать `npx medusa db:migrate` на staging | Проверенные миграции |
| Надежность | Настроить бэкап через `pg_dump` | План отката перед релизом |
| Release process | Подготовить Blue-Green rollout | Безопасное переключение трафика |
| Smoke tests | Проверить login, checkout, уведомления, историю заказов | Релиз проходит ключевые сценарии |

**Финальный checklist перед production**

- backend поднимается в production-конфигурации;
- storefront подключен к production API;
- миграции проходят без ошибок;
- бэкапы проверены;
- VK-авторизация и VK-уведомления работают на боевом домене;
- пройден end-to-end сценарий: вход -> корзина -> заказ -> уведомление.

> [!summary] Целевое состояние
> На выходе дорожной карты проект должен иметь не только работающий backend, но и полный боевой контур: авторизация, storefront, заказ, уведомления и безопасный релизный процесс.

## 4. Стратегия деплоя и обновления

### Базовый процесс

1. Собрать контейнер через `Dockerfile`.
2. Перед каждым обновлением запускать `npx medusa db:migrate`.
3. Делать резервную копию базы через `pg_dump`.

### Безопасное обновление по схеме Blue-Green

1. Поднять новую версию сервиса на соседнем порту, например `9001`.
2. Проверить работоспособность новой версии.
3. Переключить трафик в `Nginx` через `reload`.
4. Остановить и удалить старый контейнер.

> [!warning] Критичный момент
> Бэкап базы данных должен выполняться **до деплоя**, а не после. Это минимизирует риск необратимых ошибок при миграциях.

## 6. Авторизация через ВКонтакте (VK Auth)

В MedusaJS 2.0 за идентификацию отвечает `Auth Module`. Интеграция с VK позволяет реализовать вход «в один клик», что критично для конверсии в РФ.

> [!info] Техническая схема
> Используется протокол **OAuth 2.0**. Medusa выступает в роли клиента, который обменивает временный код от VK на данные пользователя.

| Компонент | Задача |
| --- | --- |
| **VK Console** | Регистрация приложения, получение `App ID` и `Service Key`. |
| **Auth Provider** | Использование библиотеки `passport-vkontakte` внутри модуля Medusa. |
| **User Mapping** | Автоматическое создание профиля `Customer` при первом входе. |
| **JWT** | Выдача стандартного токена Medusa для доступа к корзине и заказам. |

## 7. Почтовая интеграция и уведомления

Поскольку Medusa не имеет встроенного SMTP-сервера, отправка писем реализуется через `Notification Module`.

*   **Резервный канал:** Даже при входе через VK, почта необходима для отправки официальных чеков и восстановления доступа.
*   **Провайдеры:**
    *   **Для РФ:** Прямое подключение через `SMTP` или сервисы типа `UniSender` / `Mailganer`.
    *   **Глобальные:** `SendGrid`, `Resend` (лучший выбор для современных шаблонов).
*   **Событийная модель:** Вы подписываетесь на события (`order.placed`, `password.reset`) и вызываете провайдер для отправки.

## 8. Платежные решения для РФ

Для приема платежей в MedusaJS 2.0 необходимо использовать `Payment Module`. Основное требование для РФ — поддержка **54-ФЗ** (онлайн-кассы).

| Сервис | Особенности | Плагин |
| --- | --- | --- |
| **ЮKassa** | Самый популярный в РФ, поддержка всех методов (карты, СБП). | `medusa-payment-yookassa` |
| **Robokassa** | Удобен для малого бизнеса, встроенное решение по чекам. | `@gorgo/medusa-payment-robokassa` |
| **Т-Банк (Tinkoff)** | Высокая скорость выплат, отличный API. | Кастомный модуль / плагины сообщества |

> [!warning] Важно: Фискализация
> При выборе платежного модуля убедитесь, что он умеет передавать данные для чеков (товары, налоги, признаки агента) в облачную кассу.

## 9. Логистика и интеграции для РФ (legacy provider)

Для работы с российскими службами доставки в MedusaJS 2.0 рекомендуется использовать агрегаторы, чтобы избежать написания десятков отдельных интеграций.

> [!info] Что такое legacy provider?
> Это логистический шлюз, объединяющий API более 30 служб доставки (СДЭК, Почта России, Boxberry, DPD, Яндекс Доставка и др.) в один универсальный интерфейс.

| Преимущество | Описание |
| --- | --- |
| **Единый API** | Подключаете один модуль, получаете доступ ко всем службам сразу. |
| **Гибкость** | Можно включать/выключать службы в кабинете агрегатора без правки кода. |
| **Функционал** | Расчет стоимости, выбор ПВЗ на карте, печать этикеток и трекинг. |
| **Плагин** | Для Medusa доступен модуль `a third-party fulfillment module for the legacy provider`. |

## 7. Анатомия модуля в MedusaJS

В Headless-архитектуре Medusa модуль — это не одна папка с кодом, а распределенная сущность, состоящая из трех уровней:

### 1. Backend Module (Логика)
*   **Где живет:** В папке `src/modules` сервера.
*   **Что делает:** Обрабатывает данные, считает цену, общается с внешними API (например, legacy provider).
*   **UI:** Отсутствует. Взаимодействие только через JSON API.

### 2. Admin Widget (Панель управления)
*   **Где живет:** Встраивается в стандартную админку Medusa.
*   **Технология:** React.
*   **Пример:** Кнопка «Вызвать курьера СДЭК» или «Напечатать накладную» прямо в карточке заказа.

### 3. Frontend Components (Витрина)
*   **Где живет:** В папке вашего Storefront (Next.js).
*   **Технология:** React / Tailwind / Кастомные библиотеки (например, Яндекс.Карты).
*   **Что делает:** Отрисовывает выбор способа доставки для покупателя, карту пунктов выдачи (ПВЗ) и формы ввода данных.

> [!important] Важное различие
> Модуль в Medusa дает **данные и функции**, но **дизайн витрины** всегда остается за вами. Это позволяет сделать выбор СДЭК идеально вписанным в ваш уникальный дизайн сайта.

## 10. SEO и поисковая оптимизация

В Headless-архитектуре SEO на 90% реализуется на стороне **Storefront (Next.js)**, так как именно он отвечает за рендеринг HTML для поисковых роботов.

| Инструмент | Задача | Эффект |
| --- | --- | --- |
| **Next.js Metadata API** | Динамическая генерация Title, Description, OpenGraph. | Уникальные сниппеты для каждого товара. |
| **Sitemap.ts** | Автоматическая генерация `sitemap.xml` на основе API Medusa. | Быстрая индексация новых товаров. |
| **JSON-LD (Structured Data)** | Разметка товаров (цена, наличие, бренд) для Google/Яндекс. | Расширенные сниппеты (звездочки, цена в поиске). |
| **Image Optimization** | Использование `next/image` для сжатия и WebP. | Высокая скорость загрузки (PageSpeed). |
| **URL Handles** | Использование ЧПУ (человекопонятных ссылок) из Medusa. | Лучшее ранжирование по ключевым словам. |

## 11. Контент-стратегия (Блог и Новости)

Поскольку MedusaJS сфокусирована на коммерции, управление статьями и новостями реализуется через внешние или кастомные инструменты.

### Варианты реализации

| Подход | Технология | Плюсы | Минусы |
| --- | --- | --- | --- |
| **Markdown / MDX** | Локальные файлы в Next.js | **Бесплатно**, идеальное SEO, максимальная скорость. | Нужен доступ к коду для публикации. |
| **Headless CMS** | Strapi, Sanity, Contentful | Профессиональный редактор, черновики, работа с медиа. | Требует настройки еще одной системы. |
| **Custom Module** | Кастомная таблица в Medusa | **Всё в одной админке**, связь товаров со статьями. | Нужно верстать интерфейс редактора. |

> [!tip] Рекомендация для старта
> Для запуска MVP идеально подходит **Markdown/MDX**. Это не требует затрат и дает лучший результат по скорости загрузки страниц (Lighthouse). Переход на Strapi или Sanity имеет смысл при появлении выделенного контент-менеджера.
