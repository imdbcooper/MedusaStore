# Local Development

> Scope: local developer/runtime flow. This is not the production topology. Production uses [`docker-compose.prod.yml`](../docker-compose.prod.yml) with storefront, Payload, and Caddy containers.

## 1. Local architecture

Local development intentionally separates infrastructure and app runtimes:

| Part | Local default | Why |
| --- | --- | --- |
| PostgreSQL | Docker service `medusa-db` from [`docker-compose.yml`](../docker-compose.yml) | Persistent local database, predictable port mapping. |
| Redis | Docker service `medusa-redis` from [`docker-compose.yml`](../docker-compose.yml) | Runtime dependency for backend. |
| Medusa backend | Docker service `medusa-backend` from [`docker-compose.yml`](../docker-compose.yml) | Root orchestration and consistent backend dev environment. |
| Storefront | Host process via [`scripts/storefront-dev.sh`](../scripts/storefront-dev.sh) or [`scripts/storefront-start.sh`](../scripts/storefront-start.sh) | Faster Next.js dev/HMR and simpler file watching. |
| Payload CMS | Host process via [`scripts/payload-run.sh`](../scripts/payload-run.sh) | Separate Next/Payload app with guarded build/dev lifecycle. |
| Reverse proxy | Not used locally by default | Caddy is production ingress only unless you create your own local override. |

## 2. Local ports

Defaults from [`.env.example`](../.env.example):

| Variable | Default | Owner |
| --- | --- | --- |
| `POSTGRES_PORT` | `5433` | Host port mapped to `medusa-db:5432`. |
| `REDIS_PORT` | `6379` | Host port mapped to Redis. |
| `MEDUSA_BACKEND_PORT` | `9000` | Backend/API/Admin. |
| `MEDUSA_ADMIN_HMR_PORT` | `5173` | Medusa Admin Vite/HMR. |
| `STOREFRONT_PORT` | `8000` | Storefront host runtime. |
| `PAYLOAD_PORT` | `3100` | Payload host runtime. |

## 3. Clean local start

Canonical clean-clone/local flow:

```bash
cp .env.example .env
npm run bootstrap
npm run preflight
npm run dev
```

What this does:

1. `bootstrap` syncs env files, starts DB/Redis, runs Medusa migrations/seed, and writes the real storefront publishable key.
2. `preflight` checks required files, ports, and compose config.
3. `dev` starts the local root runtime: compose backend/infra plus storefront host process after backend readiness.

## 4. Local app commands

Common root commands:

| Command | Meaning |
| --- | --- |
| `npm run infra:up` | Start local PostgreSQL and Redis only. |
| `npm run backend:container:up` | Start local Medusa backend container. |
| `npm run storefront:dev` | Start storefront dev server on the host. |
| `npm run storefront:build` | Build storefront. |
| `npm run storefront:start` | Start storefront production preview on the host after build. |
| `npm run payload:dev` | Start Payload dev server on the host. |
| `npm run payload:build` | Build Payload, guarded against active dev/start process. |
| `npm run payload:seed` | Seed Payload marketing pages/globals. |
| `npm run manage` | Open interactive local management menu. |

See [`scripts/MANAGE.md`](../scripts/MANAGE.md) for menu details and local-only caveats.

## 5. Local env expectations

Root [`.env.example`](../.env.example) is the template-safe local contract. It includes placeholders and local defaults. Important local expectations:

- Replace `CHANGE_ME_*` placeholders in a real local `.env`.
- `DATABASE_URL` and `REDIS_URL` in root `.env` may use Docker service names because backend runs in Docker for the canonical local path.
- `MEDUSA_BACKEND_URL=http://localhost:9000` is fine for host-side scripts and local storefront when not running inside the production Docker network.
- `NEXT_PUBLIC_MEDUSA_PUBLISHABLE_KEY` in storefront `.env.local` is generated by bootstrap. Do not keep `REPLACE_WITH_ROOT_BOOTSTRAP` after bootstrap.
- `PAYLOAD_ENABLED=false` means content pages/news are disabled/not found in storefront, while commerce pages remain usable.
- If you enable Payload locally, set `PAYLOAD_ENABLED=true`, make sure Payload is running at `PAYLOAD_CMS_URL`, run migrations as needed, then seed content with `npm run payload:seed` if you need demo pages.

## 6. What local compose is not

[`docker-compose.yml`](../docker-compose.yml) is not production:

- no Caddy;
- no public HTTPS/ACME;
- no production storefront container;
- no production Payload container;
- local container names are `medusa-db`, `medusa-redis`, `medusa-backend`, not `medusastore-*`;
- local networking differs from production Docker-network URL expectations;
- local run does not execute GitHub Actions deploy flow;
- local smoke is not a substitute for production smoke after production infrastructure changes.

Production references live in [`docker-compose.prod.yml`](../docker-compose.prod.yml), [`Docs/architecture.md`](./architecture.md), and [`Docs/production_runbook.md`](./production_runbook.md).

## 7. Local smoke checks

After starting runtime:

```bash
npm run smoke:backend
npm run smoke:storefront
npm run smoke:browser
```

If notification smoke is needed:

```bash
npm run smoke:notification
```

For product pages, use a real product handle from the local Medusa database:

```bash
curl -I http://localhost:8000/ru/products/<real-product-handle>
```

Expected result for an existing product: `200`. Missing data should be fixed in Medusa seed/admin/data, not by adding static product params to the page.

## 8. Payload local notes

Payload runs separately from local compose. Recommended local recovery sequence:

```bash
npm run payload:status
npm run payload:stop
npm run payload:clean
npm run payload:dev
```

Do not run `npm run payload:build` while Payload dev/start is active. The build guard in [`scripts/payload-run.sh`](../scripts/payload-run.sh) exists to avoid corrupting `.next` output.

## 9. Local production-preview storefront

For browser/design verification after a production build:

```bash
npm run storefront:build
npm run storefront:start
```

This starts `next start` on `STOREFRONT_PORT` as a host process. It is useful for UI smoke but still not the production Docker/Caddy topology.
